# InterviewTraining_Identity_Api_Migrator — Миграция данных из Identity Server в БД InterviewTraining_Api

## Идея
На текущий момент это раздельные сервисы. Если их объединять - то получится каша. Если их разделять - то нет синхронизации данных. По хорошему тут надо использовать очередь. И при изменениии данных(добавление, удаление, редактирование)
на пользователей и их роли - мы должны отправлять событие в шину данных(kafka / rabbitmq) и это событие отлавливать и реагировать.
Но на текущий момент простой мигратор решает проблемы
